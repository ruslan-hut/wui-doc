# Запуск WUI як Windows service

Бінарний файл `wui.exe` має вбудовану нативну підтримку Windows service (через
[`github.com/kardianos/service`](https://github.com/kardianos/service)). Той
самий бінарний файл працює інтерактивно з термінала **і** під керуванням Windows
Service Control Manager (SCM) — обгортка (NSSM/WinSW) не потрібна.

Сервіс реєструється під:

| Поле | Значення |
|-------|-------|
| Назва сервісу (id) | `wui` |
| Відображувана назва | `WUI POS Management` |
| Опис | `WUI backend with embedded web UI for POS device management.` |

Він видимий і керований у `services.msc`, через `sc.exe`, а також через власний
прапорець `--service` бінарного файлу.

## Збірка

Крос-компіляція Windows-бінарника з macOS/Linux:

```bash
make build-windows        # -> bin/wui.exe (frontend embedded)
```

Скопіюйте `bin/wui.exe` на цільову машину (напр. `C:\WebCheck\wui.exe`).

## Встановлення / керування

Виконуйте ці команди з **підвищеними правами** (Administrator) у командному рядку. Встановлення,
видалення, запуск і зупинка сервісу потребують прав адміністратора.

```bat
:: Install (use built-in defaults)
wui.exe --service install

:: Install with an explicit config file (recommended for production)
wui.exe --config C:\WebCheck\config.yaml --service install

:: Lifecycle
wui.exe --service start
wui.exe --service stop
wui.exe --service restart
wui.exe --service status     :: -> running | stopped | unknown

:: Remove
wui.exe --service uninstall
```

`services.msc` і `sc.exe` (`sc start wui`, `sc query wui`, …) також працюють.

## Що вбудовується у сервіс

Команда `--service install` записує фіксований командний рядок, який SCM
відтворює під час кожного запуску. Дві речі встановлюються завжди, а шлях до конфігурації залежить
від того, чи передали ви `--config`:

- **`--no-browser` завжди примусово ввімкнено.** Сервіс працює в сесії 0 без
  робочого столу, тому автоматичне відкриття браузера придушується (інакше воно
  б тихо завершувалося невдало).
- **Робоча директорія = тека, що містить `wui.exe`.** Інакше SCM
  за замовчуванням використовував би `C:\Windows\System32`. Це робить відносні шляхи
  передбачуваними.

| Команда встановлення | Записаний командний рядок сервісу |
|---|---|
| `wui.exe --service install` | `wui.exe --no-browser` |
| `wui.exe --config C:\WebCheck\config.yaml --service install` | `wui.exe --config C:\WebCheck\config.yaml --no-browser` |

Коли ви встановлюєте **без** `--config`, сервіс під час запуску використовує звичайне
визначення конфігурації за замовчуванням:

1. `configs\config.yaml` відносно робочої директорії (теки exe), потім
2. змінні середовища, потім
3. вбудовані значення за замовчуванням.

Тож якщо покласти `configs\config.yaml` поруч із `wui.exe`, він буде підхоплений
автоматично. Встановлення ніколи не завершується невдало через відсутній файл конфігурації — завантаження
конфігурації переходить до запасного варіанта замість видачі помилки.

> **Порада:** для продакшну передавайте абсолютний `--config` під час встановлення, щоб
> поведінка була зафіксована й не залежала від того, де опиняться файли.

## Обліковий запис сервісу (важливо для OLE/COM)

За замовчуванням сервіс працює як **`LocalSystem`** у сесії 0 — без профілю користувача,
без робочого столу. Міст ліцензій і звітів OLE/COM (`cscript.exe`, що породжується з
`main.go`) працює в цьому контексті.

Якщо COM-об'єкту або його логіці ліцензій/звітів потрібен профіль користувача чи доступ до
робочого столу, запускайте сервіс під реальним обліковим записом користувача:

`services.msc` → **wui** → праве клацання → **Properties** → вкладка **Log On** →
**This account** → введіть Windows-користувача та пароль → перезапустіть сервіс.

## Автоматичний перезапуск при збої

kardianos не налаштовує параметри відновлення SCM. Налаштуйте їх один раз після встановлення,
щоб сервіс перезапускав сам себе у разі збою:

```bat
sc failure wui reset= 86400 actions= restart/10000/restart/10000/restart/10000
```

(Перезапуск через 10 с, до трьох разів; лічильник збоїв скидається щодоби. Зверніть увагу на
пробіли після `reset=` та `actions=` — `sc.exe` їх вимагає.)

## Журнали та усунення несправностей

- **Журнали застосунку** потрапляють туди, куди вказує `logging` у конфігурації (шлях до файлу /
  параметри ротації). Під керуванням SCM віддавайте перевагу файловому приймачу з
  абсолютним шляхом — консолі для запису немає.
- **Збої під час запуску** (напр. неможливо відкрити конфігурацію або `users.db`) повертаються
  до SCM і записуються у **Windows Event Log**
  (`Event Viewer → Windows Logs → Application`, джерело `wui`). Сервіс
  відображатиметься як такий, що не зміг запуститися, а не тихо завершиться.
- **`users.db`** розташований за шляхом `<webcheck.path>\users.db` (за замовчуванням
  `C:\ProgramData\WebCheck\users.db`). Переконайтеся, що обліковий запис сервісу може читати/писати
  в цю директорію — див. [`USERS_DB_WINDOWS_APP.md`](USERS_DB_WINDOWS_APP.md).
- **Перевірте, що він слухає:** після `--service start` відкрийте
  `http://localhost:<server.port>` (за замовчуванням `8080`) або перевірте
  `wui.exe --service status`.

## Інтерактивний режим vs. режим сервісу

Той самий шлях коду (`service.Run`) обробляє обидва:

- **Інтерактивний** (запущений з термінала): поводиться точно так само, як раніше —
  коректне завершення роботи за `Ctrl+C` / `SIGTERM`, автоматичне відкриття браузера, якщо не встановлено
  `--no-browser` або `server.no_browser`.
- **Сервіс** (запущений SCM): `Start` запускає застосунок, `Stop` виконує
  те саме коректне завершення роботи при зупинці сервісу або вимкненні машини.
  Автоматичне відкриття браузера завжди пропускається.

# Демо-режим WUI — Посібник з налаштування

Демо-режим перетворює один виділений екземпляр WUI на публічний майданчик для тестування, доступний з екрана входу для підтримки без PIN-коду. Відвідувачі натискають **Спробувати демо** і потрапляють у повноцінну сесію з правами читання/запису до даних POS демонстраційного пристрою.

**Це не лише попередній перегляд у режимі читання.** Демо-сесії передають кожне WebSocket-повідомлення — зокрема перезавантаження, запис налаштувань та замовлення ліцензій. Єдиний захист — запуск на машині, яка не містить реальних даних клієнтів. Ставтеся до демонстраційного пристрою як до кіоску: одноразового, спостережуваного, який легко переустановити.

---

## Архітектура

```
demo wui (Windows)  ──outbound WSS──>  relay  <──WSS──  browser
  role=demo                         /api/v1/support/demo/ws
  auto-reconnect                    no auth, no PIN
```

Демонстраційний екземпляр WUI реєструється на релеї під фіксованим ідентифікатором агента (за замовчуванням `demo`) і підтримує своє з'єднання з експоненційною затримкою. Точка `/api/v1/support/demo/ws` релея з'єднує анонімні браузери з цим агентом, дозволяючи до N одночасних сесій, кожна з яких обмежена жорстким TTL.

---

## 1. Релей — увімкнення демо-режиму

Відредагуйте `/etc/wui-relay/config.yaml` на хості релея та додайте блок `demo:` (показано значення за замовчуванням):

```yaml
demo:
  enabled: true
  agent_id: "demo"         # fixed ID the demo wui must register under
  max_sessions: 5          # concurrent browsers
  session_ttl: 15m         # hard per-session time limit
  rate_per_min: 10         # per-IP session-creation rate limit
```

Або використайте змінні середовища в systemd-юніті релея (`/etc/wui-relay/relay.env`):

```
WUI_RELAY_DEMO_ENABLED=true
WUI_RELAY_DEMO_AGENT_ID=demo
WUI_RELAY_DEMO_MAX_SESSIONS=5
WUI_RELAY_DEMO_SESSION_TTL=15m
WUI_RELAY_DEMO_RATE_PER_MIN=10
```

Перезапустіть релей:

```bash
sudo systemctl restart wui-relay
```

Перевірте:

```bash
curl https://web.che.ck.ua/api/v1/demo/status
# {"enabled":true,"online":false,"inUse":0,"max":5,"sessionTtlSeconds":900}
```

`online:false` — очікувана відповідь, доки ви не запустите демонстраційний пристрій WUI.

## 2. Демонстраційна машина — підготовка середовища POS

Використовуйте виділену машину Windows (фізичну або віртуальну). **Не вмикайте демо-режим на пристрої клієнта.**

1. Встановіть WebCheck та WUI так само, як під час звичайного встановлення.
2. Заповніть `testdata/` / базу даних POS **лише тестовими даними**:
   - Вигадані фіскальні номери, оператори, способи оплати.
   - Жодних реальних імен клієнтів, реальних ліцензійних ключів, реальних даних чеків.
3. Перевірте UI локально за адресою `http://localhost:8080` перед підключенням до релея.
4. Переконайтеся, що користувач, від імені якого запускається WUI, має дозвіл на запуск служби без нагляду під час завантаження.

Демонстраційний пристрій має легко переустановлюватися. Передбачте періодичні скидання, якщо відвідувачі змінюють налаштування або запускають перезавантаження.

## 3. Демонстраційна машина — налаштування WUI

Відредагуйте `configs/config.yaml` на демонстраційному пристрої:

```yaml
remote:
  enabled: true
  server_url: "wss://web.che.ck.ua/api/v1/agent/ws"
  auth:
    enabled: true
    token_secret: "<same WUI_RELAY_AUTH_TOKEN as the relay>"

demo:
  enabled: true
```

Примітки:

- `remote.auth.token_secret` має збігатися з `WUI_RELAY_AUTH_TOKEN` на релеї. Демонстраційна точка не вимагає автентифікації для браузерів, але тунель агент-релей усе одно захищений bearer-автентифікацією.
- `demo.enabled: true` автоматично змушує тунель агента підключатися під час запуску й підтримує з'єднання з експоненційною затримкою (1s → обмеження 30s, з джитером). Вам не потрібно вручну вмикати перемикач «Remote access».
- Демо-режим пригнічує генерацію PIN-кодів — інтерфейс віддаленого доступу на демонстраційному пристрої не видаватиме PIN-коди, навіть якщо ним керувати локально.

Запустіть WUI та підтвердіть з'єднання:

```bash
curl https://web.che.ck.ua/api/v1/demo/status
# {"enabled":true,"online":true,"inUse":0,"max":5,"sessionTtlSeconds":900}
```

## 4. Наскрізна перевірка

1. Відкрийте `https://web.che.ck.ua/support/login` у браузері.
2. Кнопка «Спробувати демо» має бути активною та доступною.
3. Натисніть її → ви маєте потрапити на `/registers` з даними з демонстраційного пристрою.
4. Відкрийте другий браузер — значення `inUse` збільшується, друга сесія працює незалежно.
5. Залиште сесію неактивною на 15 хвилин — вона має завершитися чистим відключенням («досягнуто ліміт часу»).
6. Завершіть процес демонстраційного WUI на машині Windows — точка статусу за лічені секунди переходить у стан `online:false`, а кнопка показує «Демо недоступне». Перезапустіть WUI — і кнопка відновлюється, щойно тунель перепідключиться.

## 5. Експлуатація демо

**Стани кнопки**, видимі відвідувачам:

| Стан | Умова | Підпис кнопки |
|---|---|---|
| ready | демо онлайн, є вільні слоти | Спробувати демо |
| full | усі `max_sessions` зайняті | Демо зайняте (N/M) |
| unavailable | демо-агент офлайн або демо вимкнено | Демо недоступне |
| loading | триває рукостискання сесії | Запуск демо… |

**Моніторинг**

- `GET /api/v1/demo/status` — дешевий знімок статусу, безпечний для опитування.
- Логи релея (`journalctl -u wui-relay -f`) записують `Demo session started` / `Demo session ended` із зазначенням причини.
- Логи демонстраційного пристрою WUI записують `Connecting to relay server` та спроби перепідключення тунелю.

**Скидання демо-даних**

Оскільки відвідувачі можуть запускати перезавантаження та операції запису, дані POS демонстраційного пристрою з часом відхилятимуться. Рекомендовані підходи:

- Зберігайте перевірений знімок бази даних POS і відновлюйте його за розкладом (cron / Task Scheduler).
- Або щотижня переустановлюйте машину з шаблону.

**Обмеження частоти запитів**

Релей обмежує частоту спроб створення сесій на `/api/v1/support/demo/ws` за IP-адресою клієнта (`rate_per_min`, за замовчуванням 10/хв). Якщо ви бачите сплески, зменшіть це значення. Встановлення його в 0 вимикає обмежувач — не рекомендовано для публічної точки.

## 6. Межі безпеки

Демо-режим свідомо жертвує контролем доступу заради простоти. Інваріанти, які забезпечує код:

- **Демо-агент ізольований від PIN-потоку.** Позначений як демо `AgentConn` відхиляє `tunnel.pin.request`, а обробник `/api/v1/support/ws` на основі PIN відмовляє будь-якому розв'язаному агенту, у якого встановлено прапорець `IsDemo`.
- **Демо-роль контролюється на релеї.** Агент може заявити `role:"demo"` лише за умови, що сам релей має `demo.enabled: true`. Інакше з'єднання закривається з `DEMO_DISABLED`.
- **Максимальна кількість сесій — жорстке обмеження.** `DemoRegistry.Start` відхиляє нові спроби з `DEMO_FULL` — без черги, без витіснення.
- **Session TTL — жорстке обмеження.** Кожна сесія має власний таймер; досягнення `session_ttl` безумовно завершує її.
- **Обмеження частоти створення сесій за IP.** Пом'якшує тривіальні зловживання, але не замінює зовнішній WAF для цілеспрямованих атак.

Від чого демо-режим **не** захищає:

- Відвідувач змінює налаштування, запускає перезавантаження або замовляє ліцензії на демонстраційному POS. Тримайте дані одноразовими.
- Відвідувач дізнається внутрішні типи повідомлень / форми протоколу. WebSocket-протокол уже є публічним у цьому репозиторії.
- Витік облікових даних із лог-файлів або даних на диску демонстраційного пристрою. Вважайте вміст диска публічним.

Якщо щось із цього важливе для вашого випадку використання, демо-режим — неправильна функція.

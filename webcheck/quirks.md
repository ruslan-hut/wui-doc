# Особливості

Поведінка, спостережена в реальній роботі, про яку не згадує посібник
постачальника. Кожен запис посилається на реальний симптом, з яким ми
зіткнулися, та обхідне рішення в нашому коді.

## 1. Пробіл перед `=` у повернутих атрибутах

Драйвер іноді видає атрибути як `Remainder ="50"` із зайвим пробілом перед
знаком рівності. Це валідний XML, але збиває наївні регулярні вирази.

**Обхідне рішення**: кожен регулярний вираз атрибута в наших парсерах
використовує `\s*=\s*`:

```go
remainderRegex = regexp.MustCompile(`(?i)Remainder\s*=\s*'([^']*)'|Remainder\s*=\s*"([^"]*)"`)
```

Зустрічається в: відповіді [`SendMessage`](send-message.md).

## 2. Суміш кодувань (UTF-8 проти Windows-1251)

Деякі виклики повертають UTF-8, деякі — Windows-1251 — правила немає. Бік
Go автовизначає для кожного виклику та декодує Win-1251 за допомогою
`charmap.Windows1251.NewDecoder()`.

```go
if !utf8.Valid(data) {
    data, _ = charmap.Windows1251.NewDecoder().Bytes(data)
}
```

Див. `parseStatusBarXML` у `internal/service/ole_windows.go`.

## 3. Hex-транспорт у VBS-воркері

Вивід `StatusBarXML` кодується в hex (4 hex-цифри на кожен Unicode-руну,
префікс `HEX=...`) воркером, бо кодова сторінка консолі Windows інакше
пошкоджує кирилицю. Бік Go декодує за допомогою `decodeHexUTF16`.

Без цього вся кирилиця в рядках чека та повідомленнях про помилки
поверталася б спотвореною.

## 4. Непослідовний регістр атрибутів для кожного методу

Глобального правила іменування немає — збігайте те, що приписує
документація для кожного методу:

| Метод             | Атрибути                            |
|-------------------|-------------------------------------|
| `GetCheck`        | `FN`, `TaxNum`, `type`              |
| `GetCheckCloudurl`| `FN`, `TaxNum`, (`type`)            |
| `SendMessage`     | `FN`, `CheckID`, `messagetype`, `Dest` |
| `AddOperator`     | `fn`, `operatorname`, `keypath`, `keypass`, `inn` (**усе в нижньому регістрі**) |
| Звіти / налаштування| `FN`, `StartDate`, `EndDate`, `Operatorid` |

Те саме значення є `TaxNum` для `GetCheck`/`GetCheckCloudurl`, але `CheckID`
для `SendMessage`. Вони **не** взаємозамінні.

## 5. Порожні атрибути можуть не пройти валідацію

Передача `Dest=""` або `type=""` повертає `Err=1004` "Неверный формат XML".
Наші білдери XML пропускають атрибути з порожніми значеннями
(`SendMessageXML` видає `Dest` лише коли він непорожній).

## 6. Нативний go-ole аварійно завершує COM DLL

`IDispatch.Invoke` у `WebCheckServer.ClassFiscal` спричиняє
ACCESS_VIOLATION. Міст VBScript обробляє ті самі виклики без проблем. Ми
зберігаємо нативний шлях go-ole лише для `GetCheckPreview` (вимкнено за
замовчуванням).

## 7. COM-драйвер не потоково-безпечний

Конкурентні виклики до одного COM-об'єкта пошкоджують внутрішній стан. Наш
воркер захищено `WindowsOLEClient.mu`, тож усі виклики серіалізуються через
єдиний процес VBScript.

## 8. Друкарська помилка в назві методу

`GetSetingsRRO` (одна `t` в "Settings"). Збігайте її точно під час виклику
COM-методу — `GetSettingsRRO` (правильне написання) не існує на драйвері.
Наша Go-обгортка названа `GetSettingsRRO` (правильно), але фактичний виклик
іде до `driver.GetSetingsRRO(xml)`.

## 9. Тривалі звіти потребують довшого таймауту

Z-звіти та звіти за період виконують фіскальне підписання та друк — вони
займають 30–60 с. Ми використовуємо `s.reportTimeout` (довший) для цих
викликів і типовий `s.timeout` (кілька секунд) для всього іншого.

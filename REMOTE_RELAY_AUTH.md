# Relay-сервер WUI — Чеклист посилення автентифікації відкритого ендпоінту

Цей документ визначає вимоги безпеки, яким має відповідати **relay-сервер**
тепер, коли агент WUI підключається через **відкритий (без токена) ендпоінт**. Механіка
розгортання (nginx, TLS, systemd) описана в [`REMOTE_RELAY_DEPLOY.md`](REMOTE_RELAY_DEPLOY.md);
цей файл охоплює лише модель авторизації та засоби контролю, що замінюють
попередньо узгоджений bearer-токен.

## Передісторія: де тепер зосереджена довіра

Агенту більше не потрібен `remote.auth.token_secret` для підключення
(`internal/service/agent_tunnel.go`). Bearer був єдиним, що обмежувало, хто
може відкрити тунель агента; без нього **ендпоінт агента є публічним**.

Авторизація доступу підтримки **не** спирається на тунель. Вона спирається на
дві видані relay, позасмугові можливості (capabilities):

1. **PIN** — генерується relay (`tunnel.pin.response`), показується
   клієнту в WUI, зачитується підтримці телефоном. Це та можливість, що
   дозволяє сесії підтримки приєднатися до конкретного з'єднання агента.
2. **SessionID** — генерується relay на початку сесії
   (`tunnel.session.start`); увесь проксійований трафік маршрутизується за ним
   (`dto.TunnelEnvelope`).

Агент не може забезпечити жоден із засобів контролю нижче — **вони цілком
є відповідальністю relay.** Relay у цьому репозиторії (`internal/relay/`) реалізує
їх, як зазначено в розділі статусу нижче.

## Статус реалізації (`internal/relay/`)

**Зроблено:**
- Ендпоінт агента приймає підключення без токена; налаштований `auth.token` усе ще
  враховується протягом міграційного вікна (`server.go checkAgentAuth`). Режим логується під час
  запуску.
- Гейт з'єднань агента по IP: обмеження одночасних + ліміт частоти нових з'єднань
  (`ip_gate.go`, підключено в `server.go handleAgent`; конфіг
  `server.max_agent_conns_per_ip`, `server.agent_rate_per_min`).
- PIN: CSPRNG (`crypto/rand`), одноразовий, TTL, заміна/відкликання по агенту, та
  **безпечна щодо глобальних колізій** генерація (`pin_registry.go`).
- Ліміт частоти введення PIN по IP + аудит-лог невдалих спроб з IP
  (`pin_registry.go AllowAttempt`, `support_conn.go ServePIN`).
- Сесії маршрутизуються за CSPRNG SessionID (UUIDv4), одна на агента, `readOnly`
  забезпечується на стороні relay (`session_manager.go`); фіскальні номери ніколи не використовуються для
  маршрутизації; кадр register обмежений (`agent_conn.go`).
- Demo-агент не адресується через потік PIN (`support_conn.go`, fail-closed).

**Налаштовується оператором / у процесі:**
- Обмеження одночасних з'єднань по IP за замовчуванням **0 (вимкнено)**, щоб не зламати
  парки за NAT-агрегацією — вмикайте його там, де клієнти не за спільним NAT.
- Ширина PIN — 6 цифр (пов'язана з UI входу підтримки `PIN_LENGTH`); розширення
  потребує узгодженої зміни фронтенду. Стійкість до перебору наразі тримається на
  лімітах частоти введення по IP + короткому TTL + одноразовості.
- Метрики/сповіщення щодо частоти з'єднань і невдалих PIN залишені на розгортання
  (логи несуть сигнал).

## Один інваріант, що не підлягає обговоренню

> **Сесії МАЮТЬ ініціюватися лише за валідованим relay PIN.**
> **Зареєстровані `fiscalNumbers` НІКОЛИ не повинні розглядатися як ідентичність чи використовуватися для
> ініціювання, маршрутизації або виявлення сесії.**

Оскільки ендпоінт агента відкритий, `fiscalNumbers` у кадрі `tunnel.register`
доступні зловмиснику для запису. Це неперевірені мітки. Захоплення (squatting) фіскального
номера жертви нешкідливе **лише доти, доки** цей інваріант дотримується — тобто PIN є
єдиною точкою входу в сесію, а фіскальні номери ніколи не виставляються як ключ маршрутизації
чи перегляду.

## Чеклист

### A. Ендпоінт агента (`/api/v1/agent/ws`) — приймати без токена, але дроселювати

- [ ] **Приймати підключення без токена.** Не відхиляйте відсутній заголовок `Authorization`.
      (За бажанням усе ще *приймайте* bearer, якщо він присутній, для міграції —
      агент надсилає його лише коли налаштовано `token_secret` — але він не повинен
      бути обов'язковим.)
- [ ] **Обмеження одночасних з'єднань по IP.** Обмежте одночасні тунелі агентів на
      одну IP-джерело (наприклад, ≤ 20). Відхиляйте підключення понад ліміт через 429/close.
- [ ] **Ліміт частоти з'єднань по IP.** Обмежте нові підключення на IP за хвилину, щоб притупити
      потоки churn/перепідключень.
- [ ] **Таймаут неактивної реєстрації.** Скидайте тунель, який підключився, але так і не надіслав
      валідний `tunnel.register` протягом N секунд.
- [ ] **Обмежений кадр register.** Обмежте довжину `fiscalNumbers` і загальний розмір кадру;
      відхиляйте надмірні реєстрації.
- [ ] **Опціональний стабільний ключ клієнта.** Використовуйте `Hostname`/`Version` із кадру register
      (і розгляньте стабільний `machineId`) як вторинний ключ ліміту частоти для
      сортування — ніколи як вхідні дані авторизації.

### B. PIN — ентропія, час життя, одноразовість

PIN тепер є останньою лінією оборони. Застарілий 6-значний PIN (простір 10⁶,
згідно з `RemoteStatus.PIN`) є **надто слабким** для відкритого ендпоінту, якщо не поєднати зі
суворим дроселюванням спроб.

- [ ] **Розширте PIN або суворо дроселюйте введення.** Віддавайте перевагу ≥ 8 символам
      однозначного base32 (~10¹²). Якщо 6-значний UX має залишитися, дроселювання спроб
      на стороні підтримки (нижче) стає обов'язковим, а не опціональним.
- [ ] **Лише CSPRNG.** Генеруйте PIN із криптографічного RNG, ніколи з PRNG чи
      позначки часу.
- [ ] **Глобальна унікальність.** Відхиляйте/перегенеровуйте при колізії з будь-яким живим PIN.
- [ ] **Короткий TTL.** ≤ 10 хвилин; враховуйте `tunnel.pin.revoke` агента.
- [ ] **Одноразовість.** Інвалідуйте PIN тієї ж миті, коли він встановлює сесію;
      PIN ніколи не повинен запускати дві сесії.
- [ ] **Обмеження запитів PIN на з'єднання.** Обмежте, скільки живих PIN може тримати одне
      з'єднання агента, щоб зловмисник не міг вичерпати простір PIN, тримаючи
      багато.

### C. Введення PIN (сторона підтримки) — стійкість до перебору

Це фактичний вектор перехоплення: зловмисний клієнт на стороні підтримки, що вгадує
активний PIN.

- [ ] **Обмежуйте частоту спроб введення PIN** на IP/обліковий запис підтримки, з експоненційним
      backoff і блокуванням.
- [ ] **Порівняння PIN за сталий час.**
- [ ] **Інвалідація при сплеску невдач.** Після K невдалих спроб проти будь-якого PIN
      розгляньте відкликання цільового PIN і сповіщення агента.
- [ ] **Без оракула.** "Невірний PIN" і "прострочений PIN" мають бути нерозрізнюваними для
      того, хто викликає, і однаково обмеженими за частотою.

### D. SessionID / токен сесії

- [ ] **CSPRNG, ≥ 128 біт** SessionID; невгадуваний, не послідовний.
- [ ] **Прив'яжіть** сесію до `{sessionId, agentConnId, readOnly, expiry}`.
- [ ] **Забезпечуйте `readOnly`** на relay для сторони підтримки — ніколи не покладайтеся на
      агента чи браузер у самообмеженні. Агент отримує `readOnly` в
      `tunnel.session.start` для відображення, але relay є воротарем.
- [ ] **Прив'язка до одного агента.** SessionID маршрутизується рівно до одного з'єднання
      агента; відхиляйте конверти, чий `sessionId` не збігається із сесією того, хто
      викликає.
- [ ] **Інвалідація при завершенні.** Скидайте токен при `tunnel.session.end`, відключенні
      агента чи закінченні TTL; розривайте парного учасника.

### E. Фіскальні номери — розглядайте як неперевірені мітки

- [ ] **Ніколи не ініціюйте за фіскальним номером.** Підтримка приєднується лише за PIN.
- [ ] **Без міжагентного виявлення.** Не виставляйте перегляд "browse agents/fiscal numbers",
      що довіряє зареєстрованим міткам.
- [ ] **Показуйте лише після сесії.** Показуйте фіскальні номери агента підтримці
      лише після встановленої за PIN сесії, чітко позначені як самозаявлені.
- [ ] **Допускайте дублікати.** Два з'єднання, що заявляють той самий фіскальний номер,
      не повинні колізіювати в маршрутизації (маршрутизація за connId/SessionID, а не фіскальним).

### F. Demo-роль

Demo-тунелі (`Role: "demo"`) навмисно анонімні та містять міст до публічних
браузерів. Тримайте їх ізольованими:

- [ ] **Один demo-слот**, виділений одноразовий агент; відхиляйте додаткові demo-
      реєстрації.
- [ ] **Без шляху PIN** для demo (агент уже придушує його).
- [ ] **Жорсткий TTL сесії + обмеження одночасності** для demo-браузерів; ніколи не маршрутизуйте demo-
      трафік до не-demo агента і навпаки.

### G. Експлуатаційне

- [ ] **Структурований аудит-лог** для: підключення/відключення агента (IP, hostname),
      видача/відкликання PIN, початок/завершення сесії та кожна невдала спроба PIN.
- [ ] **Метрики + сповіщення** для: з'єднань/IP, частоти запитів PIN, частоти невдалих PIN,
      активних сесій. Сповіщайте про аномальні сплески (сигнал перебору).
- [ ] **Жодних клієнтських даних на спокої.** Relay лише проксіює; переконайтеся, що конверти
      не логуються повністю і не зберігаються.
- [ ] **TLS термінується на проксі** (див. посібник із розгортання); агент підключається `wss://`.

## Що агент уже гарантує (тож relay не потрібно)

- Підключається лише `wss://` (TLS у транзиті).
- Надсилає bearer **лише** якщо задано `token_secret` — безпечно ігнорувати/приймати.
- Ніколи не генерує власний PIN чи SessionID — він зберігає та повертає те, що надсилає relay,
  тож усі рішення щодо ентропії/часу життя/унікальності належать relay.
- Враховує `tunnel.session.start` / `tunnel.session.end` і очищує локальний стан сесії
  при відключенні.

## Примітка про міграцію

Щоб розгорнути це без ризику дня X, тримайте relay таким, що **приймає** bearer,
роблячи його **опціональним**: агенти з токеном продовжують працювати, нові/оновлені агенти
підключаються без токена. Щойно всі агенти стануть без токена і засоби контролю вище
будуть активні, ви можете повністю вивести з ужитку `WUI_RELAY_AUTH_TOKEN`.
